# Scenario 3: Network Communication Determinism — Experiment Design

---

## 1. Problem Statement

**Research question:**
> "Does the inter-VM UDP channel provided by the EB corbos hypervisor maintain bounded and
> predictable one-way latency for a periodic safety-critical sender as the background network
> load on the low-integrity partition increases, and does packet delivery remain reliable under
> all tested load conditions?"

**Relationship to Scenarios 1 and 2:**

The three scenarios cover three distinct communication layers of the same platform:

| Scenario | Property tested            | Layer                                        |
|----------|---------------------------|----------------------------------------------|
| 1        | Freedom from interference  | Intra-VM: task timing isolation              |
| 2        | Inter-VM channel quality   | Hypervisor: shared memory ring-buffer latency |
| 3        | Network determinism        | Hypervisor: virtual Ethernet channel latency  |

Scenario 2 characterises the *shared memory* channel provided by the proxycomshm mechanism.
Scenario 3 characterises the *virtual Ethernet* channel exposed through the hypervisor's
`vnet_li_hi` virtio-net device.  These are independent communication paths: a platform could
have a predictable shared-memory channel but a non-deterministic network channel, or vice
versa.  Both properties are safety-relevant in mixed-criticality SDV architectures where
partitions must exchange data within bounded timing budgets.

**Motivation:**
In a realistic deployment, a high-integrity partition does not operate in isolation — it
publishes safety-relevant data (e.g. vehicle state, sensor readings, actuator commands) to
consumers in other partitions at a fixed rate.  If the inter-partition network channel exhibits
unpredictable latency under load, the end-to-end response time of safety functions is
undermined even if each partition individually meets its local timing requirements.

The EB corbos hypervisor exposes a virtual Ethernet link (`vnet_li_hi`) that connects vm-hi and
vm-li at the L4/Fiasco.OC layer.  Packets traverse QEMU's virtio-net backend and are delivered
through the guest kernel's standard IP/UDP stack.  This scenario characterises how that channel
behaves for a 1 ms periodic UDP sender on vm-hi as background UDP flood traffic on vm-li
increases.

---

## 2. Variables

### Independent variables

**Primary — Background flood rate N (Mbps):**
Sustained UDP flood traffic generated by `net_flood` on vm-li and directed at vm-hi over the
same `vnet_li_hi` interface that carries the safety-critical flow.  This creates bidirectional
contention on the shared virtual NIC and its host-side virtio backend.

Load levels: **N = 0, 16, 32, 64, 128, 256 Mbps**

- N = 0: baseline — only the periodic sender runs; no background flood.
- N ≥ 128 Mbps: expected to approach or exceed QEMU virtio-net saturation on this platform.

Rationale for UDP flooding: UDP best-effort traffic is the worst-case contender for the
safety-critical flow because it generates no back-pressure and can consume all available
bandwidth without yielding.

**Secondary — Network impairment via `tc netem` (N = 0 only):**
Applied to vm-li's vnet_li_hi interface (ingress and egress) to emulate adverse network
conditions independently of flood load.

| Condition | `tc netem` parameters                          |
|-----------|------------------------------------------------|
| D0        | none (baseline, same as C0)                    |
| D1        | `delay 1ms`                                    |
| D2        | `delay 5ms 1ms distribution normal`            |
| L1        | `loss 0.5%`                                    |
| L2        | `loss 2%`                                      |

Note: applying tc netem requires iproute2 to be present in vm-li.  If not already in the base
image, add `iproute2` to `li/root.yaml` packages.

### Dependent variables (metrics)

- **Approximate one-way latency (OWL, µs)** — `recv_time_ns − send_time_ns` where
  `send_time_ns` is stamped on vm-hi immediately before `sendto()` and `recv_time_ns`
  is stamped on vm-li immediately after `recvfrom()` returns.  In QEMU, vm-hi and vm-li
  share the same host ARM virtual counter through the hypervisor, so clock drift between
  the two VMs is negligible (< 1 µs); see Section 9 for caveats.
- **Inter-arrival jitter (IAJ, µs)** — `|IAT − T_period|` where IAT is the measured
  inter-arrival time between consecutive packets at vm-li and T_period = 1 ms is the nominal
  sender period.  This metric does not require any cross-VM clock assumption and is the
  primary indicator of temporal predictability at the receiver.
- **Tail latency** — P95, P99, P99.9 of both OWL and IAJ.  Tail growth is the
  safety-relevant metric.
- **Packet loss rate (%)** — proportion of expected packets not received within a 500 ms
  timeout, detected via gaps in the sequence number field.
- **Flood effective throughput (Mbps)** — measured by `net_flood` to confirm that the
  configured load level was actually achieved.

## 3. Hypotheses

**H₀ (null):** Background flood rate N has no statistically significant effect on OWL or IAJ.
The hypervisor's VM partitioning and the virtio-net backend fully isolate the safety-critical
sender from vm-li's flood traffic.

**H₁:** Mean OWL and IAJ increase monotonically with N.  As the flood rate grows, the shared
virtio-net backend and QEMU's host-side packet scheduling are increasingly contested, causing
the safety-critical packets to be queued or delayed before delivery.

**H₂ (tail growth):** P99.9 OWL grows faster than mean OWL as N increases.  Occasional severe
delays caused by burst queuing or QEMU vCPU preemption produce a heavy tail that is the most
safety-relevant indicator of channel degradation.

**H₃ (reliability):** Packet delivery remains 100% reliable (0% loss) at all flood load
levels.  The safety-critical sender uses a dedicated UDP socket and the receiver's stop-and-
collect protocol; if no loss occurs, the channel can be declared reliable under the tested
conditions.  If loss does occur, the threshold flood rate N* at which loss first appears is
reported.

**H₄ (impairment fidelity):** Applied `tc netem` delay and loss are faithfully reproduced in
the measured OWL and loss rate respectively.  If tc netem adds D ms of delay, the mean OWL
increase should be approximately D ms.  This validates the measurement infrastructure and
confirms that the clock alignment assumption holds.

## 4. Experimental Setup

### Platform architecture and required changes

The platform is unchanged from Scenarios 1 and 2 (EB corbos hypervisor on QEMU ARM virt
machine, vm-hi on vCPUs 2–3, vm-li on vCPUs 0–1) with **two configuration changes**:

**Change 1 — IP reconfiguration of the inter-VM NIC.**
The `vnet_li_hi` virtual NIC is already defined in `hv/virt-common.dtsi` and present in both
VMs.  vm-hi's kernel boot parameter currently assigns it `192.168.7.100/24`, which conflicts
with vm-li's DHCP-assigned address on the external QEMU user network (also `192.168.7.0/24`).
For Scenario 3 the inter-VM subnet is moved to `10.0.1.0/24`:

- vm-hi: `10.0.1.1/24` (static, via kernel `ip=` boot parameter in `hv/hv-base.yaml`)
- vm-li: `10.0.1.2/24` (static, configured in `li/config_root.sh` startup script)

The external QEMU user network (`192.168.7.0/24`) used for SSH access is unchanged.

**Change 2 — vm-hi init binary.**
`hv/hv-base.yaml` is updated to replace `init=/speedometer` (Scenario 1) or `init=/shm_echo`
(Scenario 2) with `init=/net_sender`.  `net_sender` is cross-compiled and installed into the
hi root filesystem by the CMake build, and `hi/config_root.sh` is updated to mark it
executable.

No DTS or hypervisor topology changes are required.

### Network topology

```
┌──────────────────────────────────────────────────────┐
│                  EB corbos Hypervisor                │
│                                                      │
│   ┌─────────────────────┐   ┌──────────────────────┐ │
│   │     vm-hi (ASIL)    │   │    vm-li  (QM)       │ │
│   │   vCPUs 2–3         │   │   vCPUs 0–1          │ │
│   │   10.0.1.1          │   │   10.0.1.2           │ │
│   │                     │   │   192.168.7.2 (SSH)  │ │
│   │  net_sender         │   │  net_receiver        │ │
│   │  (1 ms UDP, →)      │   │  + net_flood (←)     │ │
│   └──────────┬──────────┘   └──────────┬───────────┘ │
│              │    vnet_li_hi (virtio)   │             │
│              └──────────────────────────┘             │
└──────────────────────────────────────────────────────┘
                         │
              QEMU virtio-net (user net)
                         │
                 Host (SSH 192.168.7.5)
```

- Safety-critical flow: `net_sender` (vm-hi) → `net_receiver` (vm-li), UDP port 5000,
  1 ms period, 64 B packets.
- Background flow: `net_flood` (vm-li) → vm-hi UDP port 5001, configurable Mbps,
  1400 B packets (near MTU to maximise bandwidth pressure).  vm-hi does not listen on
  port 5001; the kernel drops arriving packets after IP delivery.  The primary effect
  is contention for the shared virtio-net backend on both directions of the link.

### Sender timing model

`net_sender` uses `clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME, ...)` to achieve a 1 ms
periodic wakeup, identical in structure to the speedometer loop in Scenario 1.  As with
Scenario 1, `sched_setscheduler(SCHED_FIFO)` is expected to fail on the hi VM safety kernel
(returning `ENOSYS`); the sender therefore runs as `SCHED_OTHER`.  Scheduling jitter in the
sender contributes to the measured OWL and IAJ — this is an acknowledged limitation (Section 9)
and a faithful reflection of the platform's actual behaviour.

`send_time_ns` is stamped immediately before `sendto()` to minimise the sender-side contribution
to OWL.  `recv_time_ns` is stamped immediately after `recvfrom()` returns on vm-li.

### Clock alignment

In QEMU, both VMs read the ARM generic timer counter (`CNTPCT_EL0`) which is driven by the
same host `CLOCK_MONOTONIC` source.  The hypervisor virtualises the counter without offset, so
vm-hi and vm-li clocks are expected to agree within < 1 µs.  The OWL estimate
`recv_time_ns − send_time_ns` is therefore meaningful in this environment.  This assumption
is explicit and does **not** hold on real distributed hardware; see Section 9.

### Message structure

```c
/* Defined in scenario3/common/common.h */
struct net_message {
    uint64_t seq;              /* Sequence number (0-based, monotonically increasing) */
    uint64_t send_time_ns;     /* CLOCK_MONOTONIC on vm-hi immediately before sendto() */
    uint8_t  payload[NET_PAYLOAD_BYTES]; /* Padding; default 48 B → 64 B total struct  */
};
```

Default `NET_PAYLOAD_BYTES = 48` gives a 64 B struct (8 + 8 + 48), a clean multiple of 8 with
no compiler padding.  At 1 ms interval this produces 512 kbps of safety-critical traffic,
negligible relative to the flood rates tested.

### Experiment matrix

**Primary (varying N, fixed payload = 64 B, 5000 packets/run):**

| Condition | N (Mbps) | `net_receiver` invocation                       | Runs |
|-----------|----------|-------------------------------------------------|------|
| C0        | 0        | `net_receiver -n 5000`                          | 10   |
| C1        | 16       | `net_receiver -n 5000 -B 16`                    | 10   |
| C2        | 32       | `net_receiver -n 5000 -B 32`                    | 10   |
| C3        | 64       | `net_receiver -n 5000 -B 64`                    | 10   |
| C4        | 128      | `net_receiver -n 5000 -B 128`                   | 10   |
| C5        | 256      | `net_receiver -n 5000 -B 256`                   | 10   |

At 1 ms period, 5000 packets = 5 s of measurement per run, giving 50 s per condition across
10 runs (plus ~3 min QEMU boot overhead per condition).

**Secondary — network impairment (N = 0, tc netem applied on vm-li's vnet_li_hi interface,
1000 packets/run):**

| Condition | Impairment                               | `tc netem` command                         | Runs |
|-----------|------------------------------------------|--------------------------------------------|------|
| D0        | none (same as C0)                        | —                                          | —    |
| D1        | +1 ms fixed delay                        | `delay 1ms`                                | 5    |
| D2        | +5 ms delay, 1 ms jitter (normal)        | `delay 5ms 1ms distribution normal`        | 5    |
| L1        | 0.5% random loss                         | `loss 0.5%`                                | 5    |
| L2        | 2.0% random loss                         | `loss 2%`                                  | 5    |

netem is applied with:
```bash
tc qdisc add dev <vnet_li_hi_iface> root netem <params>
```
and removed after each condition with:
```bash
tc qdisc del dev <vnet_li_hi_iface> root
```

**Total: 60 primary + 20 secondary = 80 runs.**  QEMU is rebooted between primary conditions
for a clean state.  Within a condition, multiple runs share a single QEMU boot (net_sender
runs continuously as PID 1).  Secondary conditions also each get a fresh QEMU boot; netem is
applied after SSH becomes available and before the first run.

## 5. New Applications

### `net_sender` (vm-hi — PID 1 for this scenario)

A minimal periodic UDP sender.  It has no measurement logic — its sole function is to send
a precisely timed stream of packets and run indefinitely as PID 1.

Behaviour:
1. Opens a UDP socket bound to `10.0.1.1:5000`; sets `SO_PRIORITY` if available.
2. Establishes `sendto()` target as `10.0.1.2:5000`.
3. Tries `sched_setscheduler(SCHED_FIFO, 90)` — logs success or failure; continues either way.
4. Loops indefinitely at 1 ms using `clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME)`:
   - Stamps `send_time_ns = clock_gettime(CLOCK_MONOTONIC)`.
   - Fills `net_message` with `seq`, `send_time_ns`, and a fixed payload byte (`0xAB`).
   - Calls `sendto()`.
   - Increments `seq`.
5. Prints a progress line every 10,000 packets (every 10 s).
6. Never exits; if running as PID 1, remains in the send loop.

Console output (captured in `qemu_serial.log` by the experiment script):
```
net_sender: started, target=10.0.1.2:5000, period=1ms
net_sender: RT scheduling: FAILED (ENOSYS) — running as SCHED_OTHER
net_sender: [10000 pkts] seq=9999 elapsed=10.003s
net_sender: [20000 pkts] seq=19999 elapsed=20.007s
...
```

### `net_receiver` (vm-li)

A UDP receiver with integrated background flood control.

Behaviour:
1. Opens a UDP socket bound to `0.0.0.0:5000`.
2. Optionally spawns `net_flood` as a background subprocess targeting `10.0.1.1:5001`
   at the requested Mbps rate.  Waits 2 s for the flood to ramp up before measurements begin.
3. Collects `n_messages` packets using `recvfrom()`:
   - Records `recv_time_ns = clock_gettime(CLOCK_MONOTONIC)` immediately after return.
   - Computes `owlatency_ns = recv_time_ns − msg.send_time_ns`.
   - Computes `iat_ns = recv_time_ns − prev_recv_time_ns` (skipped for the first packet).
   - Detects loss as gaps in `msg.seq`; increments loss counter for each missing sequence.
   - Records per-packet data in a statically allocated array (up to `MAX_SAMPLES = 10000`).
4. Kills `net_flood` subprocess with SIGTERM; waits for it to exit.
5. Prints statistics summary; writes per-packet CSV.

Command-line interface:
```
net_receiver [options]
  -n <count>    Packets to collect (default: 5000, max: 10000)
  -B <Mbps>     Background flood rate in Mbps (default: 0 = disabled)
  -o <file>     Output CSV path (default: net_receiver_log.csv)
  -h            Show this help
```

Console output (captured as `run_XX.txt` by the experiment script):
```
========== NET RECEIVER RESULTS ==========
Packets expected:   5000
Packets received:   5000
Packets lost:          0 (0.000%)
OWL (us):
  Mean:     48.3    Std dev:  12.1
  Min:       8.4    Max:     412.7
  P50:      45.2    P95:      71.3
  P99:     122.6    P99.9:   394.1
IAJ (us):
  Mean:     14.7    Std dev:   9.8
  Min:       0.1    Max:     321.5
  P50:      11.2    P95:      33.9
  P99:      87.4    P99.9:   295.6
==========================================
```

CSV columns: `seq, send_time_ns, recv_time_ns, owlatency_us, iat_us, lost`

### `net_flood` (vm-li)

A minimal UDP flooding tool.

Behaviour:
1. Opens a UDP socket; calls `connect()` to target IP:port (default `10.0.1.1:5001`).
2. Pre-fills a 1400 B payload buffer.
3. Sends packets in a tight loop, throttling via `nanosleep` to achieve the target Mbps rate.
   Rate control: sleep interval = `(pkt_size * 8) / target_bps` seconds per packet.
4. Runs until SIGTERM or `--timeout` expires.
5. Prints total bytes sent and effective throughput on exit.

Command-line interface:
```
net_flood [options]
  -d <ip>       Destination IP (default: 10.0.1.1)
  -p <port>     Destination port (default: 5001)
  -b <Mbps>     Target bandwidth in Mbps (default: 64)
  -s <bytes>    Packet size in bytes (default: 1400)
  -t <sec>      Timeout in seconds (0 = run until SIGTERM, default: 0)
  -h            Show this help
```

## 6. Run Procedure

1. Build the Scenario 3 image: `net_sender` built for vm-hi (AArch64, safety kernel), placed
   in hi root; `net_receiver` and `net_flood` built for vm-li (AArch64, standard kernel),
   placed in li root.  Run `task image` to produce `build/image.raw`.
2. Start QEMU (`task qemu`); wait for boot and SSH readiness.
3. `net_sender` starts automatically on vm-hi (PID 1) and begins sending at 1 ms.
4. SSH into vm-li; wait 5 s for `net_sender` to be fully up and sending.
5. *For primary conditions:* invoke `net_receiver -n 5000 [-B <N>]`.
6. `net_receiver` discovers the running flood (if -B > 0), waits 2 s for ramp-up, collects
   5000 packets, stops flood, prints results.
7. Output captured to `run_XX.txt`; CSV saved to `run_XX.csv`.
8. Repeat steps 5–7 for remaining runs of this condition (no QEMU reboot between runs of
   the same condition).
9. Stop QEMU; reboot for the next condition.
10. *For secondary (netem) conditions:* after SSH readiness, apply the netem rule with `tc`,
    then invoke `net_receiver -n 1000`.  Remove the netem rule after all runs of this
    condition are complete.

## 7. Acceptance Criteria

**Reliability criterion:** packet loss rate = 0% at all load levels (H₃).

**Predictability criteria** at flood rate N:
- P99 OWL ≤ 5× baseline (C0) P99 OWL.
- IAJ standard deviation ≤ 3× baseline IAJ standard deviation.

The **channel degradation threshold N\*** is the smallest N for which at least one
predictability criterion is violated.  If no criterion is violated up to N = 256 Mbps, the
channel is reported as "predictable across all tested load levels."

**Impairment fidelity criterion** (H₄): for condition D1 (+1 ms netem delay), the increase
in mean OWL relative to D0 must be in the range [0.8 ms, 1.2 ms] (within ±20% of the
configured delay).

---

## 8. Data Analysis and Hypothesis Testing

### For H₀ (no effect of N):
- Kruskal-Wallis test on per-run mean OWL across all 6 primary conditions.
- If p < 0.05, reject H₀ and quantify the effect.

### For H₁ (monotonic latency increase):
- Spearman rank correlation (ρ) between N and per-run mean OWL and IAJ standard deviation.
- Confirm H₁ if ρ > 0.8 and p < 0.05 for at least one metric.
- Line plot of mean OWL ± std vs N, and IAJ std vs N.

### For H₂ (tail growth faster than mean):
- Compute ratio P99.9 / mean OWL at each load level.
- If this ratio increases with N, the tail grows disproportionately — report the growth factor.
- Multi-line plot of mean / P99 / P99.9 OWL on the same axis vs N.

### For H₃ (zero loss):
- Report total lost packets per condition and per run.
- If any loss occurs: report the condition, run index, and first lost sequence number.
  Determine whether it was a send-buffer overflow, receive-buffer overflow, or a genuine
  drop in the virtual network.

### For H₄ (impairment fidelity):
- For D1 and D2: compare the increase in mean OWL (relative to D0) to the configured
  netem delay.
- For L1 and L2: compare measured loss rate to configured loss rate; a close match
  validates that `tc netem` is correctly shaping the vnet_li_hi interface.

### Visualisations:
- Line plot of mean OWL ± std vs N (primary trend, H₁)
- Multi-line plot of P50 / P99 / P99.9 OWL vs N (tail growth, H₂)
- Box plots of per-run mean OWL per condition (run-to-run variance)
- Line plot of IAJ standard deviation vs N (channel jitter)
- Bar chart of packet loss rate per condition (expected: all zero, H₃)
- Bar chart of mean OWL per impairment condition (D0–D2, H₄)
- Bar chart of measured vs configured loss rate for L1, L2 (H₄)

## 9. Platform Constraints and Limitations

**QEMU virtio-net does not model real automotive Ethernet hardware.**  The `vnet_li_hi`
device is a host-process socket pair mediated by QEMU's virtio-net backend.  There is no
modelled PHY, transmission wire, switch fabric, or AXI memory bus.  Latency values reflect
host OS thread scheduling and QEMU's packet handling, not physical network timing.
On real EBcLfSA hardware, inter-VM Ethernet traffic would traverse physical NICs and a
hardware switch, introducing propagation delays, MAC scheduling, and potential queuing not
present in QEMU.

**No TSN guarantees.**  This scenario does not claim or test Time-Sensitive Networking
properties.  The virtual network has no hardware clock synchronisation (IEEE 802.1AS), no
credit-based shaper (IEEE 802.1Qav), and no time-aware shaper (IEEE 802.1Qbv).  Results
cannot be extrapolated to TSN deployments.

**Clock alignment is a QEMU-specific property.**  The OWL estimates depend on vm-hi and vm-li
sharing the same physical time source through the hypervisor's virtual ARM counter.  On real
hardware, establishing sub-microsecond clock alignment between VMs requires hardware PTP or
IEEE 802.1AS, which are not available in this environment.  On real hardware, only receiver-
side inter-arrival jitter (IAJ) can be computed without additional synchronisation
infrastructure.

**No SCHED_FIFO on vm-hi.**  As established in Scenario 1, the hi VM safety kernel blocks
`sched_setscheduler` (returns `ENOSYS`).  `net_sender` therefore runs as `SCHED_OTHER`.
Scheduling jitter in the sender itself contributes to the measured OWL and IAJ and cannot
be separated from network-path variability without a hardware timestamp at the NIC egress
point, which is unavailable in QEMU.  Measured latency is therefore a combined upper bound
on sender-scheduling jitter plus network-path latency.

**Flood target drops arrive at vm-hi.**  The background flood packets are addressed to
`10.0.1.1:5001` on vm-hi, which has no socket bound to that port.  The guest kernel will
generate ICMP Port Unreachable responses unless silenced with an iptables rule, introducing
a small additional load on vm-hi's network stack.  For strict isolation of the effect on
the forward path (vm-hi → vm-li), this overhead should be measured in the baseline and
subtracted, or suppressed with `iptables -I INPUT -p udp --dport 5001 -j DROP` in the hi
VM init script.

**Flood rate accuracy.**  `net_flood`'s rate control is software-only (`nanosleep`-based
throttling).  Actual throughput may deviate from the requested N by 10–20% depending on
the host scheduler.  `net_flood` reports its effective throughput on exit; runs where the
effective rate deviates by more than 25% from the target should be flagged.

## 10. Expected Outcomes and Conclusions Framework

Given QEMU's virtio-net and host-thread scheduling model:

- **H₁ is expected to be confirmed:** as N grows, the virtio-net backend becomes increasingly
  busy processing flood packets in both directions, delaying timely delivery of the 1 ms
  safety-critical packets to vm-li.  Mean OWL and IAJ should increase.
- **H₂ is expected to be confirmed:** occasional severe queuing or QEMU vCPU preemption
  events will produce a heavy OWL tail, particularly at N ≥ 128 Mbps where the virtual NIC
  approaches saturation.
- **H₃ is expected to hold at low-to-moderate N:** at N = 0 to 64 Mbps, the channel should
  remain loss-free.  At N = 128–256 Mbps, receiver socket buffer overflow or QEMU internal
  drops may cause non-zero loss; if so, N* is reported.
- **H₄ is expected to hold:** QEMU propagates `tc netem` rules faithfully; measured OWL
  increases should closely track the configured netem delay.

**Conclusions to draw:**
- Report N* or state "no degradation threshold within N = 0–256 Mbps."
- Quantify OWL growth: "at N = X Mbps, P99 OWL was Y µs compared to Z µs at baseline —
  a W× increase."
- Report whether the channel is reliable under all tested load conditions.
- Relate to the safety argument: if the worst-case P99.9 OWL across all tested conditions
  is X µs, any safety function relying on this channel must budget at least X µs of
  communication overhead in its worst-case execution time analysis.
- Compare with Scenario 2: if the shared-memory channel (Scenario 2) exhibits lower and
  more bounded latency than the network channel (Scenario 3), this provides quantitative
  justification for using shared memory rather than IP networking for high-criticality
  inter-partition communication in EBcLfSA deployments.

**Potential follow-up:**
- Repeat on real EBcLfSA hardware for physically meaningful latency and bandwidth values.
- Test with `SO_PRIORITY` and `IP_TOS`/DSCP marking combined with a priority-based qdisc
  on vm-li to evaluate whether traffic differentiation can protect the safety-critical flow.
- Add a concurrent speedometer RT loop in vm-hi (Scenario 1 style) to measure whether
  network packet processing in vm-hi's kernel interferes with the RT task's timing.
- Test with pipeline-mode sending (multiple outstanding packets) to measure maximum
  sustained throughput of the safety-critical channel.
