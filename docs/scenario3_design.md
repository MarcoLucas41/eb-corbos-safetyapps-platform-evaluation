# Scenario 3: Network Channel — Experiment Design

---

## 1. Problem Statement

**Relationship to Scenarios 1 and 2:**

The three scenarios cover three distinct communication layers of the same platform:

| Scenario | Property tested           | Layer                               |
|----------|---------------------------|-------------------------------------|
| 1        | Task Isolation 		| Intra-VM: task timing isolation     |
| 2        | Shared Memory Channels  | Hypervisor: shared memory latency   |
| 3        | Network channel       | Hypervisor: virtual Ethernet latency|

Scenario 2 characterises the *shared memory* channels (`hv_proxycomshm` and `hv_hicomshm`).
Scenario 3 characterises the *virtual Ethernet* channel exposed through the hypervisor's
`vnet_li_hi` virtio-net device.  These are independent communication paths: a platform could
have a predictable shared-memory channel but a non-deterministic network channel, or vice
versa.  Both properties are safety-relevant in mixed-criticality SDV architectures where
partitions must exchange data within bounded timing budgets.

**Motivation:**
In a realistic deployment, a high-integrity partition does not operate in isolation — it
publishes safety-relevant data (e.g. vehicle state, sensor readings, actuator commands) to
consumers in other partitions at a fixed rate.  If the inter-partition network channel exhibits
unpredictable latency under load, the end-to-end response time of safety functions is
undermined even if each partition individually meets its local timing requirements.

The EB corbos hypervisor exposes a virtual Ethernet link (`vnet_li_hi`) that connects vm-hi and
vm-li at the L4/Fiasco.OC layer.  Packets traverse QEMU's virtio-net backend and are delivered
through the guest kernel's standard IP/UDP or TCP stack.

This scenario is implemented as **two complementary variants** that test different network paths:

| Variant | Network path | Measurement side | Research question |
|---------|-------------|------------------|-------------------|
| A — cross-VM | vm-li ↔ vm-hi via `vnet_li_hi` | vm-li | Does the inter-VM network channel maintain bounded RTL as background flood load increases? |
| B — HI-only | vm-hi loopback (127.0.0.1) | vm-hi | Does vm-li load affect the vm-hi network stack even when all traffic stays local to vm-hi? |

Variant A characterises the cross-VM virtual Ethernet channel under direct network contention
from a UDP flood.  Variant B isolates the *indirect* interference path — both ping and echo
processes run on vm-hi, communicating over the local loopback interface, which vm-li has no
access to.  If latency increases under vm-li flood load even when all traffic is local to vm-hi,
this implicates host-level resource contention (QEMU scheduler, shared CPU cache, virtio-net
backend) rather than direct network path contention.  Together, the two variants decompose the
observed RTL degradation into attributable causes.

Variant A has **two build configurations** (normal and reversed_udp) that are functionally
identical — both implement the same ping-echo protocol — but use different source file layouts
and binary names.  The normal build is the primary one; the reversed_udp build exists as a
separate deployment target for build-system flexibility.

## 2. Variables

### Independent variables (shared across both variants)

**Primary — Background flood rate N (Mbps):**
Sustained UDP flood traffic generated by `net_flood` on vm-li and directed at vm-hi over the
same `vnet_li_hi` interface that carries the safety-critical flow.  This creates bidirectional
contention on the shared virtual NIC and its host-side virtio backend.

Load levels: **N = 0, 16, 32, 64, 128, 256 Mbps**

- N = 0: baseline — only the periodic ping-echo runs; no background flood.
- N ≥ 128 Mbps: expected to approach or exceed QEMU virtio-net saturation on this platform.

Rationale for UDP flooding: UDP best-effort traffic is the worst-case contender for the
safety-critical flow because it generates no back-pressure and can consume all available
bandwidth without yielding.

**Secondary — Transport protocol:**
UDP (default) or TCP.  TCP adds kernel-level flow control, Nagle coalescing (disabled via
`TCP_NODELAY`), and connection state.  Both protocols are negotiated per-run via the
handshake/trigger mechanism.

**Tertiary — Network impairment via `tc netem` (Variant A, N = 0 only):**
Applied to vm-li's vnet_li_hi interface to emulate adverse network conditions independently of
flood load.

| Condition | `tc netem` parameters                          |
|-----------|-------------------------------------------------|
| D0        | none (baseline, same as C0)                     |
| D1        | `delay 1ms`                                     |
| D2        | `delay 5ms 1ms distribution normal`             |
| L1        | `loss 0.5%`                                     |
| L2        | `loss 2%`                                       |

Note: applying tc netem requires iproute2 to be present in vm-li.

### Dependent variables (metrics, both variants)

- **Round-trip latency (RTL, µs)** — `recv_time_ns − send_time_ns`, where both timestamps are
  taken on the pinger's `CLOCK_MONOTONIC`.  In Variant A, both timestamps are on vm-li; in
  Variant B, both are on vm-hi.  No cross-VM clock synchronisation required in either case.
- **Inter-arrival jitter (IAJ, µs)** — `|IAT − T_period|` where IAT is the measured
  inter-arrival time between consecutive echo responses and T_period = 1 ms is the nominal
  probe period.
- **Tail latency** — P95, P99, P99.9 of both RTL and IAJ.  Tail growth is the
  safety-relevant metric.
- **Packet loss rate (%)** — proportion of probes that timed out (100 ms threshold for
  echo response), detected via sequence mismatch.
- **Flood effective throughput (Mbps)** — measured by `net_flood` to confirm the configured
  load level was actually achieved.

### Cross-variant comparison metric

**RTL ratio (hicom / cross-VM) at each N** — if vm-li flood load affects both paths equally,
this ratio stays constant; if the HI-only path is less affected, the ratio decreases with N,
quantifying how much of the degradation in Variant A is caused by cross-VM network contention
versus indirect host-level interference.

## 3. Hypotheses

### Variant A — cross-VM

**H₀ (null):** Background flood rate N has no statistically significant effect on RTL or IAJ.
The hypervisor's VM partitioning and the virtio-net backend fully isolate the safety-critical
ping-echo from vm-li's flood traffic.

**H₁:** Mean RTL and IAJ increase monotonically with N.  As the flood rate grows, the shared
virtio-net backend and QEMU's host-side packet scheduling are increasingly contested, causing
the safety-critical packets to be queued or delayed before delivery.

**H₂ (tail growth):** P99.9 RTL grows faster than mean RTL as N increases.  Occasional severe
delays caused by burst queuing or QEMU vCPU preemption produce a heavy tail that is the most
safety-relevant indicator of channel degradation.

**H₃ (reliability):** Packet delivery remains 100% reliable (0% loss) at all flood load
levels.

**H₄ (impairment fidelity):** Applied `tc netem` delay and loss are faithfully reproduced in
the measured RTL and loss rate respectively.  This validates the measurement infrastructure.

### Variant B — HI-only (hicom)

**H₅ (isolation):** RTL on the HI-only loopback path is significantly less sensitive to N than
on the cross-VM path.  Because the flood traffic traverses the inter-VM NIC while Variant B's
ping-echo stays on vm-hi's loopback, any degradation must arise through indirect paths only
(QEMU host scheduler contention, shared CPU cache pollution).  If H₅ is confirmed, the network
path isolation between VMs extends beyond the NIC itself.

**H₅a (stronger form):** RTL on the HI-only path does not increase significantly with N.  This
would indicate that QEMU's host-level resource sharing has negligible impact on the vm-hi
loopback stack at the tested flood levels.

**H₅b (weaker form):** RTL on the HI-only path does increase with N, but less steeply than on
the cross-VM path.  This would indicate that indirect interference (host scheduler / cache) is
present but smaller than the direct network contention tested in Variant A.

## 4. Experimental Setup

### Platform architecture

- Hypervisor: EB corbos Linux (L4/Fiasco.OC-based), running on QEMU ARM virt machine
- CPU: Emulated ARM Cortex-A57 (ARMv8), 4 vCPUs total
- vm-hi: vCPUs 2–3 (cpus: 0xc), 128 MB RAM
- vm-li: vCPUs 0–1 (cpus: 0x3), 512 MB RAM

### IP configuration

The `vnet_li_hi` virtual NIC is defined in `hv/virt-common.dtsi` and present in both VMs.
The inter-VM subnet is `10.0.1.0/24`:

- vm-hi: `10.0.1.1/24` (static, via kernel `ip=` boot parameter)
- vm-li: `10.0.1.2/24` (static, configured in `li/config_root.sh`)

The external QEMU user network (`192.168.7.0/24`) used for SSH access is unchanged.

### Measurement method: Round-Trip Latency (RTL)

All variants use a **ping-echo model**.  The pinger timestamps `send_time_ns` on its own
`CLOCK_MONOTONIC` just before send, receives the echo, timestamps `recv_time_ns` just after
recv, and computes `RTL = recv_time_ns − send_time_ns`.  Both timestamps come from the same
clock, so no cross-VM synchronisation is required.

Probes are sent at a 1 ms period using `nanosleep`-based absolute-time scheduling.

### Network topology

```
┌─────────────────────────────────────────────────────────┐
│                  EB corbos Hypervisor                    │
│                                                         │
│   ┌───────────────────────┐   ┌───────────────────────┐ │
│   │     vm-hi (ASIL)      │   │    vm-li  (QM)        │ │
│   │   vCPUs 2–3           │   │   vCPUs 0–1           │ │
│   │   10.0.1.1            │   │   10.0.1.2            │ │
│   │                       │   │   192.168.7.2 (SSH)   │ │
│   │ ┌───────────────────┐ │   │                       │ │
│   │ │ Variant A:        │ │   │  Variant A:           │ │
│   │ │  echo server      │◄├───┤► pinger + net_flood   │ │
│   │ │  (net_sender)     │ │   │  (net_receiver)       │ │
│   │ └───────────────────┘ │   │                       │ │
│   │ ┌───────────────────┐ │   │  Variant B:           │ │
│   │ │ Variant B:        │ │   │  net_trigger          │ │
│   │ │  hi_net_ping ◄──► │ │   │  (UDP trigger only)   │ │
│   │ │  hi_net_echo      │ │   │  + net_flood          │ │
│   │ │  (loopback:5002)  │ │   │                       │ │
│   │ └───────────────────┘ │   │                       │ │
│   └───────────┬───────────┘   └───────────┬───────────┘ │
│               │    vnet_li_hi (virtio)     │             │
│               └────────────────────────────┘             │
└─────────────────────────────────────────────────────────┘
```

- **Variant A safety-critical flow:** vm-li (pinger) ↔ vm-hi (echo), UDP/TCP port 5000,
  1 ms period, 64 B packets.  RTL measured on vm-li.
- **Variant B safety-critical flow:** hi_net_ping ↔ hi_net_echo, both on vm-hi,
  UDP/TCP 127.0.0.1:5002, 1 ms period, 64 B packets.  RTL measured on vm-hi.
- **Background flow (both variants):** `net_flood` (vm-li) → vm-hi UDP port 5001,
  configurable Mbps, 1400 B packets.  vm-hi has no listener on port 5001; an iptables
  rule suppresses ICMP port-unreachable replies.

### Message structure

```c
/* Shared across all variants (common/common.h in each build) */
struct net_message {
    uint64_t seq;              /* Sequence number (0-based per run)     */
    uint64_t send_time_ns;     /* CLOCK_MONOTONIC on pinger before send */
    uint8_t  payload[NET_PAYLOAD_BYTES]; /* Default 48 B → 64 B total  */
};
```

Default `NET_PAYLOAD_BYTES = 48` gives a 64 B struct (8 + 8 + 48).  At 1 ms interval this
produces 512 kbps of ping-echo traffic, negligible relative to the flood rates tested.

### Protocol negotiation

**Variant A — HELLO handshake (always over UDP):**
1. Pinger sends HELLO (seq=`NET_CTRL_SEQ_HELLO`, payload[0]=protocol, payload[1..8]=n_probes).
2. Echo server replies with HELLO_ACK.
3. If TCP: pinger opens TCP connection; echo server accepts.

**Variant B — trigger + HELLO:**
1. `net_trigger` (vm-li) sends TRIGGER (seq=`NET_CTRL_SEQ_TRIGGER`, payload[0]=protocol,
   payload[1..8]=n_probes) to `hi_net_ping` via UDP on NET_HI_IP:5000.
2. `hi_net_ping` replies TRIGGER_ACK, then sends HELLO to `hi_net_echo` (127.0.0.1:5002).
3. `hi_net_echo` replies HELLO_ACK.  If TCP: local TCP connection established.

## 5. Applications

### Shared utility

#### `net_flood` (vm-li)

Minimal UDP flooding tool, shared across all variants.

Behaviour:
1. Opens a UDP socket; `connect()` to target IP:port (default `10.0.1.1:5001`).
2. Pre-fills a 1400 B payload buffer.
3. Sends packets in a tight loop, throttling via `nanosleep` to achieve the target Mbps rate.
4. Runs until SIGTERM or `--timeout` expires.
5. Prints total bytes sent and effective throughput on exit.

CLI: `net_flood [-d ip] [-p port] [-b Mbps] [-s bytes] [-t sec] [-h]`

### Variant A — cross-VM (two build configurations)

#### Normal build (`scenario3/`)

**`net_sender`** (vm-hi — PID 1, echo server):
Waits for a HELLO from the pinger on vm-li, negotiates UDP/TCP transport, then echoes every
received probe back unchanged.  Supports multiple runs per QEMU boot — on silence timeout
(3 s) or TCP close, re-binds and waits for the next HELLO.

**`net_receiver`** (vm-li — pinger):
Sends periodic 1 ms probes to the echo server on vm-hi and measures RTL on its own
`CLOCK_MONOTONIC`.  Optionally spawns `net_flood` as a background subprocess before measurement.
Records per-probe RTL and IAJ.  Prints statistics summary; writes per-packet CSV.

CLI: `net_receiver [-n count] [-B Mbps] [-t] [-o csv] [-h]`

Console output format:
```
========== NET RECEIVER (PINGER) RESULTS ==========
Probes sent:        5000
Echoes received:    5000
Lost (no echo):        0 (0.000%)
RTL (us):
  Mean:    XXX.X    Std dev: XXX.X
  Min:     XXX.X    Max:     XXX.X
  P50:     XXX.X    P95:     XXX.X
  P99:     XXX.X    P99.9:   XXX.X
IAJ (us):
  Mean:    XXX.X    Std dev: XXX.X
  ...
===================================================
```

CSV columns: `seq, send_time_ns, recv_time_ns, rtlatency_us, iat_us`

#### Reversed UDP build (`scenario3/reversed_udp/`)

Functionally identical to the normal build.  Provides a separate build target with different
binary names for deployment flexibility.

**`net_receiver_hi`** (vm-hi — PID 1, echo server): same echo logic as `net_sender`.

**`net_sender_li`** (vm-li — pinger): same ping/RTL logic as `net_receiver`.

CLI: `net_sender_li [-n count] [-B Mbps] [-t] [-o csv] [-h]`

### Variant B — HI-only (`scenario3/hicom_variant/`)

#### `hi_net_ping` (vm-hi — PID 1)

Pinger and statistics collector.  On startup: spawns `hi_net_echo` as a child process via
`clone(CLONE_VFORK | CLONE_VM)` + `execve`, then creates a UDP socket on `NET_HI_IP:5000`
for receiving triggers from vm-li.  Prints `HI_NET_PING_READY` once ready, then waits for a
`NET_CTRL_SEQ_TRIGGER` message from `net_trigger`.  On trigger: performs HELLO handshake with
`hi_net_echo` (127.0.0.1:5002), runs N probes at 1 ms period, collects RTL and IAJ samples,
prints results, prints `HI_NET_PING_DONE run=N`.  Repeats for subsequent triggers without
rebooting.

#### `hi_net_echo` (vm-hi — child of `hi_net_ping`)

Echo server co-located in vm-hi.  Binds to 127.0.0.1:5002 (UDP), waits for HELLO from
`hi_net_ping`, negotiates UDP/TCP, echoes every received probe unchanged.  Supports multiple
runs — on silence timeout or TCP close, re-binds and waits for the next HELLO.

#### `net_trigger` (vm-li)

Run trigger.  Sends a single UDP `NET_CTRL_SEQ_TRIGGER` message to `hi_net_ping` on
`NET_HI_IP:5000`, specifying the protocol (UDP/TCP) and probe count.  Optionally starts
`net_flood` as a background process before sending the trigger.  Waits for the experiment
to complete before stopping the flood.

CLI: `net_trigger [-n count] [-t] [-B Mbps] [-h]`

Console output format for Variant B (extracted from QEMU serial log):

## 6. Run Procedure

### Variant A — cross-VM

1. Build hi image with `net_sender` as PID 1 (`init=/net_sender` in `hv-base.yaml`).
   Build li image with `net_receiver`, `net_flood`.  Run `task image`.
2. Boot QEMU (`task qemu`); wait for li VM SSH readiness.
3. `net_sender` starts automatically on vm-hi and waits for a HELLO.
4. SSH into vm-li; invoke `net_receiver -n 5000 [-B <N>] [-t]`.
5. `net_receiver` performs HELLO handshake, optionally starts `net_flood` (waits 2 s ramp-up),
   runs ping-echo loop, stops flood, prints results.
6. Output captured to `run_XX.txt`; CSV saved to `run_XX.csv`.
7. Repeat steps 4–6 for remaining runs (no QEMU reboot within a condition).
8. Stop QEMU; reboot for the next condition.
9. *For netem conditions:* after SSH readiness, apply netem rule with `tc`, run
   `net_receiver -n 1000`, then remove netem after all runs.

### Variant B — HI-only

1. Build hi image with `hi_net_ping` as PID 1 (`init=/hi_net_ping`).
   Both `hi_net_ping` and `hi_net_echo` must be in the hi VM root filesystem.
   Build li image with `net_trigger`, `net_flood`.
2. Boot QEMU; wait for li VM SSH readiness **and** for `HI_NET_PING_READY` in the
   vm-hi serial console log.
3. `hi_net_ping` starts, spawns `hi_net_echo`, prints `HI_NET_PING_READY`.
4. SSH into vm-li; invoke `net_trigger -n 5000 [-t] [-B <N>]`.
5. `net_trigger` optionally starts `net_flood` (waits 2 s), sends TRIGGER, waits for
   experiment to complete, stops flood.
6. Wait for `HI_NET_PING_DONE run=R` in serial log; extract result block to `run_XX.txt`.
7. Repeat steps 4–6 for remaining runs.
8. Stop QEMU; reboot for the next condition.

Results for Variant B are extracted from the QEMU serial log between the
`HI_NET_PING_START`/`HI_NET_PING_DONE` markers.

## 7. Acceptance Criteria

### Variant A — cross-VM

**Reliability:** packet loss rate = 0% at all load levels (H₃).

**Predictability** at flood rate N:
- P99 RTL ≤ 5× baseline (C0) P99 RTL.
- IAJ standard deviation ≤ 3× baseline IAJ standard deviation.

The **channel degradation threshold N\*** is the smallest N for which either predictability
criterion is violated.  If neither is violated up to N = 256 Mbps, the channel is reported
as "predictable across all tested load levels."

**Impairment fidelity (H₄):** for condition D1 (+1 ms netem delay), the increase in mean RTL
relative to D0 must be in the range [0.8 ms, 1.2 ms] (within ±20% of the configured delay).

### Variant B — HI-only

**Reliability:** packet loss rate = 0% at all load levels.

**Isolation criterion (H₅):** The slope of mean RTL vs N in Variant B is statistically
significantly smaller than in Variant A (paired Wilcoxon test on per-condition medians,
p < 0.05).

**Strong isolation (H₅a):** Kruskal-Wallis test on Variant B per-run mean RTL across all
6 conditions fails to reject H₀ (p ≥ 0.05) — flood rate has no significant effect on
HI-only loopback RTL.

## 8. Data Analysis and Hypothesis Testing

### Per-variant tests (applied to both A and B independently)

**H₀ / H₅a (no effect of N):**
- Kruskal-Wallis test on per-run mean RTL across all 6 conditions.
- Reject if p < 0.05.

**H₁ / H₅b (monotonic RTL increase):**
- Spearman rank correlation (ρ) between N and per-run mean RTL and IAJ standard deviation.
- Confirm if ρ > 0.8 and p < 0.05.

**H₂ (tail growth):**
- Compute ratio P99.9 / mean at each N for Variant A.
- If this ratio increases with N, the tail grows disproportionately — report the growth factor.

**H₃ (zero loss):**
- Report total lost probes per condition per variant.
- If any loss: report condition, run index, and first lost sequence number.

**H₄ (impairment fidelity, Variant A only):**
- For D1 and D2: compare the increase in mean RTL (relative to D0) to the configured
  netem delay.
- For L1 and L2: compare measured loss rate to configured loss rate.

### Cross-variant comparison (H₅)

- **Overlay plot:** mean RTL vs N for both Variant A and Variant B on the same axes.
  The vertical gap between curves at each N represents the contribution of cross-VM network
  path contention to RTL.
- **Slope comparison:** fit a linear trend to mean RTL vs N for each variant; compare slopes
  and 95% confidence intervals.  Non-overlapping CIs confirm H₅.
- **RTL ratio (B/A) vs N:** if isolation is effective, this ratio should approach 1 or
  decrease as N increases (HI-only path becomes relatively less affected).
- Paired Wilcoxon test on per-condition median RTL: Variant A vs Variant B at each N.

### Visualisations

- Line plot of mean RTL ± std vs N (both variants overlaid — primary comparison)
- Multi-line plot of P50 / P99 / P99.9 RTL vs N (Variant A — tail growth, H₂)
- Box plots of per-run mean RTL per condition (both variants, side-by-side)
- Line plot of IAJ standard deviation vs N (channel jitter, both variants)
- Bar chart of packet loss rate per condition and variant
- RTL ratio (Variant B / Variant A) vs N (isolation effectiveness)
- Bar chart of mean RTL per impairment condition (D0–D2, Variant A — H₄)
- Bar chart of measured vs configured loss rate for L1, L2 (H₄)
- UDP vs TCP comparison: mean RTL ± std vs N for both protocols within each variant

## 9. Platform Constraints and Limitations

**QEMU virtio-net does not model real automotive Ethernet hardware.**  The `vnet_li_hi`
device is a host-process socket pair mediated by QEMU's virtio-net backend.  There is no
modelled PHY, transmission wire, switch fabric, or AXI memory bus.  Latency values reflect
host OS thread scheduling and QEMU's packet handling, not physical network timing.

**No TSN guarantees.**  This scenario does not claim or test Time-Sensitive Networking
properties.  The virtual network has no hardware clock synchronisation (IEEE 802.1AS), no
credit-based shaper (IEEE 802.1Qav), and no time-aware shaper (IEEE 802.1Qbv).  Results
cannot be extrapolated to TSN deployments.

**No SCHED_FIFO on vm-hi.**  As established in Scenario 1, the hi VM safety kernel blocks
`sched_setscheduler` (returns `ENOSYS`).  In Variant A, the echo server on vm-hi runs as
`SCHED_OTHER`; in Variant B, both `hi_net_ping` and `hi_net_echo` run as `SCHED_OTHER`.
The vm-li pinger (Variant A) uses SCHED_FIFO where available.

**The HI-only isolation result is QEMU-specific.**  In QEMU, vm-li's `net_flood` competes
with vm-hi's processes for *host OS* CPU time and shared QEMU internal structures.  If H₅a
holds (no effect on HI-only RTL), this only demonstrates that the QEMU scheduler does not
measurably perturb the HI partition at these flood levels — not that the hypervisor provides
physical network isolation.  On real hardware with shared L3 cache and physical NIC DMA,
vm-li's flood traffic would more directly compete with vm-hi's network stack.

**No cross-VM clock synchronisation needed.**  RTL is measured as a round-trip on a single
clock.  Both Variant A (vm-li) and Variant B (vm-hi) use their own `CLOCK_MONOTONIC`.
Absolute RTL values are *not directly comparable* across variants (different network paths),
but the slope-vs-N analysis is meaningful for both.

**Flood target drops arrive at vm-hi.**  Background flood packets addressed to
`10.0.1.1:5001` have no listener.  An iptables rule in `hi_config_root.sh` suppresses ICMP
Port Unreachable responses to avoid perturbing the measured link.

**Flood rate accuracy.**  `net_flood`'s rate control is software-only (`nanosleep`-based
throttling).  Actual throughput may deviate from the requested N by 10–20%.  `net_flood`
reports its effective throughput on exit; runs where the effective rate deviates by more than
25% should be flagged.

**Variant B loopback assumption.**  `hi_net_echo` listens on 127.0.0.1:5002.  The safety
kernel must support the loopback interface.  If loopback is not available, `NET_ECHO_IP` in
`hicom_variant/common/common.h` can be changed to `NET_HI_IP` ("10.0.1.1") — traffic to a
local IP is still routed locally by the kernel without traversing the physical interface.

## 10. Expected Outcomes and Conclusions Framework

### Variant A — cross-VM

- **H₁ expected confirmed:** as N grows, the virtio-net backend becomes increasingly busy
  processing flood packets, delaying timely delivery of the ping-echo probes.
- **H₂ expected confirmed:** occasional severe queuing or QEMU vCPU preemption events will
  produce a heavy RTL tail, particularly at N ≥ 128 Mbps.
- **H₃ expected to hold at low-to-moderate N:** at N = 0 to 64 Mbps, the channel should
  remain loss-free.  At N = 128–256 Mbps, socket buffer overflow may cause non-zero loss.
- **H₄ expected to hold:** QEMU propagates `tc netem` rules faithfully.

### Variant B — HI-only

- **H₅a may or may not hold** depending on QEMU host load.  If the host machine has spare
  cores, vm-li's `net_flood` may not visibly delay vm-hi's loopback processing.
- **H₅b is expected to be confirmed:** since vm-li's flood traffic does not traverse vm-hi's
  loopback path, the dominant delay mechanism in Variant A (cross-VM packet delivery) is absent.

### Conclusions to draw (both variants)

- Report N* for Variant A or state "no degradation threshold within N = 0–256 Mbps."
- Quantify RTL growth: "at N = X Mbps, cross-VM P99 RTL was Y µs (W× baseline); HI-only
  P99 RTL was Z µs (V× baseline)."
- Report the isolation gap: "Y − Z µs of RTL degradation at N = X is directly attributable
  to cross-VM network contention."
- Compare UDP vs TCP: report whether TCP's kernel-level flow control provides better or worse
  determinism under flood load.
- Relate to the safety argument: budget the worst-case P99.9 RTL of the chosen channel and
  protocol as communication overhead in any safety function's WCET analysis.
- Compare with Scenario 2: if the shared-memory channel (Scenario 2) exhibits lower and more
  bounded latency than the network channel (Scenario 3), this provides quantitative
  justification for using shared memory rather than IP networking for high-criticality
  inter-partition communication in EBcLfSA deployments.

**Potential follow-up:**
- Repeat on real EBcLfSA hardware for physically meaningful latency and bandwidth values.
- Test with `SO_PRIORITY` and `IP_TOS`/DSCP marking combined with a priority-based qdisc
  to evaluate whether traffic differentiation can protect the safety-critical flow.
- Add a concurrent speedometer RT loop in vm-hi (Scenario 1 style) to measure whether
  network packet processing interferes with the RT task's timing.
- Test with pipeline-mode sending (multiple outstanding packets) to measure maximum
